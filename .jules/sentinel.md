## 2024-05-24 - [Auth] Prevent Password Oracle in Registration
**Vulnerability:** The registration endpoint (`/api/v1/auth/register`) was using `authenticate_user(db, email, password)` to check if a user existed before creating a new account.
**Learning:** Using an authentication function to check for existence creates a password oracle / timing vulnerability. If the email exists and the attacker guesses the wrong password, the request takes longer (due to password hashing validation via `verify_password`). This timing difference leaks whether an email is registered or not, allowing for user enumeration. It also causes a 500 Server Error if the wrong password is provided for an existing user instead of properly rejecting it with a 409 Conflict.
**Prevention:** We added a lightweight `get_user_by_email` function that queries the database without checking the password. We updated the registration logic to use this instead, securely verifying if the email exists without introducing timing variations.
