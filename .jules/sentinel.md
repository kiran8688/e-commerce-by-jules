## 2025-02-28 - [Authentication Bypass via Password Oracle]
**Vulnerability:** The `register` endpoint used `authenticate_user` to check if a user existed by providing the new user's email and password. This allowed an attacker to determine if an email address was registered by trying to register with the same email and a wrong password. Since `authenticate_user` checks the password hash, if the user didn't exist, it would return `None` (meaning "doesn't exist" or "wrong password"), and the system would create a new user. If the user *did* exist and the wrong password was used, it would also return `None` (wrong password) and attempt to create a new user with that email, likely failing at the database level with a unique constraint violation. However, if the attacker guessed the *correct* password, `authenticate_user` would return the user, and the endpoint would return a `409 Conflict`. This creates a password oracle where an attacker can brute-force the password of an existing user during registration.
**Learning:** Checking for user existence using authentication functions that also verify passwords creates a side-channel where the success/failure depends on the password correctness, exposing a password oracle.
**Prevention:** Always use dedicated functions (e.g., `get_user_by_email`) that only perform simple database lookups without password verification when checking for user existence.
