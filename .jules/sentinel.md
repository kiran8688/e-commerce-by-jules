## 2026-04-30 - [Fix password oracle vulnerability in registration endpoint]
**Vulnerability:** The `/register` endpoint was using `authenticate_user` (which verifies both email and password) to check if a user already exists. If the password provided during registration happened to match the existing user's password, the endpoint would return a "User already exists" error. If the password did not match, it would fail the password check and return a 409 error anyway, but with different timing or potentially different error paths, creating a password oracle that could allow an attacker to guess passwords of existing users via timing or error discrepancies.
**Learning:** Checking user existence should only rely on unique identifiers (like email) and never involve validating sensitive credentials like passwords, to avoid inadvertent side channels or password oracles.
**Prevention:** Use a dedicated lookup function, like `get_user_by_email`, which only queries the database for the user's email without interacting with the stored password hashes.
